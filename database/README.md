# Database Setup for VictoriaMetrics Jobs

This directory contains the database setup files for VictoriaMetrics jobs integration.

## Files

- `database_ddl.sql` - PostgreSQL DDL script for extraction job tables
- `vm_forecast_ddl.sql` - PostgreSQL DDL script for metrics forecast storage
- `vm_extracted_metrics_labels_migration.sql` - Migration script to add label storage to extractor
- `setup_database.py` - Python script to automate database setup
- `README.md` - This documentation file

## Quick Setup

### Fresh Installation

For a fresh installation, run the DDL scripts in order:

```bash
# 1. Create extraction tables
psql -h localhost -U your_user -d your_db -f database/database_ddl.sql

# 2. Create forecast table
psql -h localhost -U your_user -d your_db -f database/vm_forecast_ddl.sql
```

### Migration from Old Schema

If you have an existing `vm_extracted_metrics` table without label storage:

```bash
# Run the migration script to add metric_labels column and rename audit_id
psql -h localhost -U your_user -d your_db -f database/vm_extracted_metrics_labels_migration.sql
```

**Note**: The migration renames `audit_id` to `metric_auid`. If you have views or foreign keys referencing this column, you'll need to drop and recreate them.

### Option 1: Automated Setup (Recommended)

Use the Python setup script for automated database creation:

```bash
# Basic setup with defaults
python setup_database.py

# Setup with custom database
DB_NAME=my_apex_db DB_USER=my_user DB_PASSWORD=secret python setup_database.py

# Setup with remote database
DB_HOST=db.example.com DB_PORT=5432 DB_NAME=apex_prod DB_USER=apex_user DB_PASSWORD=secret python setup_database.py
```

### Option 2: Manual Setup

1. Create the database:
```sql
CREATE DATABASE victoria_metrics_jobs;
```

2. Connect to the database:
```bash
psql -h localhost -U postgres -d victoria_metrics_jobs
```

3. Execute the DDL script:
```sql
\i database_ddl.sql
```

## Database Schema

The VictoriaMetrics jobs integration uses the `vm_jobs` schema with the following tables:

### Core Tables

- **`vm_extraction_jobs`** - Tracks extraction job executions from VictoriaMetrics to database
- **`vm_extracted_metrics`** - Stores extracted metrics data from VictoriaMetrics with JSONB label storage
- **`vm_forecast`** - Stores Prophet-generated metric forecasts (partitioned by source)

### Extracted Metrics Table Details (vm_extracted_metrics)

The `vm_extracted_metrics` table stores historical metrics extracted from VictoriaMetrics by the extractor job:

**Columns:**
- `id` (BIGSERIAL) - Auto-incrementing primary key
- `biz_date` (DATE) - Business date of the metric
- `metric_auid` (VARCHAR) - Application unique identifier (renamed from audit_id)
- `metric_name` (VARCHAR) - Name of the metric
- `value` (DECIMAL) - Metric value
- `timestamp` (TIMESTAMP) - Original timestamp from VictoriaMetrics
- `metric_labels` (JSONB) - Additional metric labels as JSON (excludes: source/job, auid, biz_date, __name__)
- `extracted_at` (TIMESTAMP) - When the metric was extracted
- `job_id` (VARCHAR) - ID of the extraction job
- `job_execution_timestamp` (TIMESTAMP) - Timestamp of the job execution

**Indexes:**
- `idx_vm_extracted_metrics_labels` - GIN index for JSONB queries
- `idx_vm_extracted_metrics_auid` - B-tree index on metric_auid

**Label Storage Pattern:**
The extractor job stores standard labels in dedicated columns (source from job label, auid, biz_date) and stores all remaining labels as JSONB in the `metric_labels` column for flexible querying.

### Forecast Table Details (vm_forecast)

The `vm_forecast` table stores time-series forecasts generated by the `metrics_forecast` job using Prophet:

**Columns:**
- `source` (VARCHAR) - Source job name (e.g., 'apex_collector')
- `biz_date` (DATE) - Business date being forecasted (future date)
- `metric_auid` (VARCHAR) - Application unique ID from the metric
- `metric_name` (VARCHAR) - Name of the metric being forecasted
- `metric_value` (NUMERIC) - Forecasted value
- `metric_labels` (JSONB) - Additional metric labels as JSON
- `forecast_type` (VARCHAR) - Type of forecast (trend, lower, upper)
- `created_at` (TIMESTAMP) - When the forecast was first created (client-supplied)
- `updated_at` (TIMESTAMP) - When the forecast was last updated (client-supplied)

**Primary Key:** (source, biz_date, metric_auid, metric_name, forecast_type)

**Partitioning:** The table is partitioned by LIST on the `source` column for better performance with multiple data sources.

- `vm_forecast_autosys_jobs` - Dedicated partition for source='autosys_jobs'
- `vm_forecast_default` - Default partition for all other sources

To create additional source-specific partitions:
```sql
CREATE TABLE vm_forecast_your_source_name 
  PARTITION OF vm_forecast FOR VALUES IN ('your_source_name');
```

**Indexes:**
- `idx_forecasts_biz_date` - Query by business date
- `idx_forecasts_metric_name` - Query by metric name
- `idx_forecasts_auid` - Query by application UID
- `idx_forecasts_forecast_type` - Query by forecast type
- `idx_forecasts_metric_labels` - GIN index for JSONB queries
- `idx_forecasts_source_date_name` - Composite index for common patterns

### Views

- **`v_extraction_job_summary`** - Summary of extraction job executions
- **`v_metrics_summary_by_date`** - Metrics summary grouped by business date


## Configuration

Update your configuration file with the database connection details:

```yaml
database:
  host: localhost
  port: 5432
  name: victoria_metrics_jobs
  user: vm_user
  password: vm_password
  ssl_mode: prefer
  connect_timeout: 10
  application_name: victoria_metrics_jobs
  schema: vm_jobs
```

## Permissions

The DDL script grants permissions to a `vm_user` role. Adjust as needed:

```sql
-- Create user if it doesn't exist
CREATE USER vm_user WITH PASSWORD 'vm_password';

-- Grant permissions
GRANT USAGE ON SCHEMA vm_jobs TO vm_user;

GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA vm_jobs TO vm_user;
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA vm_jobs TO vm_user;
```

## Monitoring Queries

### Check Job Status
```sql
-- Replace 'your_schema' with your actual schema name
SELECT * FROM your_schema.v_extraction_job_summary 
WHERE business_date >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY started_at DESC;
```


### Check Metrics Summary
```sql
-- Replace 'your_schema' with your actual schema name
SELECT * FROM your_schema.v_metrics_summary_by_date 
WHERE business_date >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY business_date DESC, metric_name;
```

### Check for Failed Jobs
```sql
-- Replace 'your_schema' with your actual schema name
SELECT * FROM your_schema.extraction_jobs 
WHERE status = 'failed' 
AND started_at >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY started_at DESC;
```

### Check Recent Forecasts
```sql
-- Replace 'your_schema' with your actual schema name
-- Get recent forecasts for a specific metric
SELECT 
    source,
    biz_date,
    metric_name,
    metric_auid,
    forecast_type,
    metric_value,
    updated_at
FROM your_schema.vm_forecast
WHERE metric_name = 'your_metric_name'
  AND biz_date >= CURRENT_DATE
ORDER BY biz_date, forecast_type;
```

### Check Forecast Coverage
```sql
-- Replace 'your_schema' with your actual schema name
-- See forecast coverage by source and date
SELECT 
    source,
    biz_date,
    COUNT(DISTINCT metric_name) as metric_count,
    COUNT(*) as forecast_count,
    MAX(updated_at) as last_updated
FROM your_schema.vm_forecast
WHERE biz_date >= CURRENT_DATE
GROUP BY source, biz_date
ORDER BY source, biz_date;
```

### Query Extracted Metrics with Labels
```sql
-- Replace 'your_schema' with your actual schema name
-- Query metrics with specific label values
SELECT 
    biz_date,
    metric_name,
    metric_auid,
    value,
    metric_labels,
    extracted_at
FROM your_schema.vm_extracted_metrics
WHERE metric_labels @> '{"environment": "production"}'::jsonb
  AND biz_date >= CURRENT_DATE - INTERVAL '7 days'
ORDER BY biz_date DESC, metric_name
LIMIT 100;
```

### Check Label Distribution
```sql
-- Replace 'your_schema' with your actual schema name
-- See what labels are being stored
SELECT 
    metric_name,
    COUNT(*) as record_count,
    COUNT(DISTINCT metric_auid) as unique_auids,
    jsonb_object_keys(metric_labels) as label_key
FROM your_schema.vm_extracted_metrics
WHERE biz_date >= CURRENT_DATE - INTERVAL '7 days'
GROUP BY metric_name, label_key
ORDER BY metric_name, label_key;
```

## Maintenance

### Cleanup Old Data
```sql
-- Replace 'your_schema' with your actual schema name
-- Clean up old extraction jobs (keep 90 days)
DELETE FROM your_schema.extraction_jobs 
WHERE started_at < NOW() - INTERVAL '90 days';

-- Clean up old extracted metrics (keep 1 year)
DELETE FROM your_schema.extracted_metrics 
WHERE extracted_at < NOW() - INTERVAL '1 year';

-- Clean up old forecasts (keep forecasts for past 30 days only)
DELETE FROM your_schema.vm_forecast 
WHERE biz_date < CURRENT_DATE - INTERVAL '30 days';
```


## Troubleshooting

### Common Issues

1. **Permission Denied**: Ensure the database user has proper permissions on the schema and tables
2. **Schema Not Found**: Make sure the `apex_integration` schema exists
3. **Connection Failed**: Check database host, port, and credentials
4. **Table Not Found**: Verify the DDL script was executed successfully

### Verification Queries

```sql
-- Check if tables exist (replace 'your_schema' with your actual schema name)
SELECT table_name FROM information_schema.tables WHERE table_schema = 'your_schema' AND table_name IN ('extraction_jobs', 'extracted_metrics');

-- Check if views exist (replace 'your_schema' with your actual schema name)
SELECT table_name FROM information_schema.views WHERE table_schema = 'your_schema' AND table_name IN ('v_extraction_job_summary', 'v_metrics_summary_by_date');
```

## Performance Considerations

### Indexes

The DDL script creates comprehensive indexes for optimal performance:

- Primary key indexes on all tables
- Foreign key indexes
- Business date indexes for time-based queries
- GIN indexes on JSONB columns
- Composite indexes for common query patterns

### Partitioning

For high-volume environments, consider partitioning the `extracted_metrics` table by `business_date`:

```sql
-- Example partitioning by month
CREATE TABLE apex_integration.extracted_metrics_2024_01 
PARTITION OF apex_integration.extracted_metrics
FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
```

### Vacuum and Analyze

Regular maintenance:

```sql
-- Analyze tables for query optimization
ANALYZE your_schema.vm_extracted_metrics;
ANALYZE your_schema.vm_extraction_jobs;
ANALYZE your_schema.vm_forecast;

-- Vacuum to reclaim space
VACUUM ANALYZE your_schema.vm_extracted_metrics;
VACUUM ANALYZE your_schema.vm_forecast;
```

## Security

### Best Practices

1. **Use dedicated database user** with minimal required permissions
2. **Enable SSL** for remote connections
3. **Regular password rotation** for database users
4. **Monitor access logs** for suspicious activity
5. **Backup regularly** including schema and data

### Connection Security

```yaml
database:
  ssl_mode: require  # Force SSL for production
  ssl_cert: /path/to/client-cert.pem
  ssl_key: /path/to/client-key.pem
  ssl_rootcert: /path/to/ca-cert.pem
```

## Backup and Recovery

### Backup Schema and Data
```bash
# Backup schema only
# Replace 'your_schema' with your actual schema name
pg_dump -h localhost -U postgres -d victoria_metrics_jobs --schema=your_schema --schema-only > vm_jobs_schema.sql

# Backup data only
pg_dump -h localhost -U postgres -d victoria_metrics_jobs --schema=your_schema --data-only > vm_jobs_data.sql

# Full backup
pg_dump -h localhost -U postgres -d victoria_metrics_jobs --schema=your_schema > vm_jobs_full.sql
```

### Restore
```bash
# Restore from backup
psql -h localhost -U postgres -d victoria_metrics_jobs < vm_jobs_full.sql
```
